# Contributing to the Knowledge Base

Thank you for your interest in contributing to the Service Design × Business Architecture × Journey Operations knowledge base!

## 📋 Ways to Contribute

### 1. Suggest New Research Topics

Have a topic that should be covered? Open an issue with:
- **Topic title** — Clear, specific name
- **Why it matters** — Relevance to the knowledge base scope
- **Suggested sources** — 2-3 starting points for research
- **Target audience** — Who would benefit most

### 2. Correct or Update Existing Content

If you find errors, outdated information, or missing nuances:
- Open an issue describing the correction needed
- Or submit a pull request with the fix
- Include source citations for any factual updates

### 3. Add Case Studies

Real-world examples strengthen the knowledge base. Contributions should include:
- **Organization** (or anonymized description)
- **Context and challenge** faced
- **Approach taken** — methods, frameworks, tools used
- **Outcome or insight** — quantified where possible
- **Source** — link or citation

Format: Add to the relevant topic document under a "Case Studies" section, or contribute to the synthesis document's Case References section.

### 4. Add Practitioner Perspectives

Quotes, insights, or lessons learned from practitioners are valuable:
- Include attribution (name, role, organization)
- Provide context for when/where the insight originated
- Link to source if publicly available

### 5. Translate Content

Help make this knowledge accessible in other languages:
- Create translated versions in a `/translations/{language-code}/` folder
- Maintain the same document structure
- Note translation date and translator

## 📝 Content Guidelines

### Style

- **Write for practitioners** — Actionable, clear, minimal jargon
- **Show, don't just tell** — Use examples, tables, diagrams where helpful
- **Cite sources** — Every factual claim should trace to a reference
- **Distinguish evidence levels**:
  - *Established practice* — Widely adopted, multiple converging sources
  - *Emerging pattern* — Growing adoption, limited validation
  - *Theoretical/aspirational* — Conceptually sound, early adoption only

### Structure

Each research topic document should include:
1. **Overview** — What the topic covers and why it matters
2. **Core Concepts** — Key definitions and principles
3. **Frameworks & Standards** — Established methodologies
4. **Practical Application** — How to use in practice
5. **Retail Context** — Specific considerations for retail organizations
6. **Notable Practitioners** — Key people and organizations
7. **Sources** — Complete reference list with URLs

### Formatting

- Use Markdown
- Tables for comparisons (keep under 6 columns for readability)
- Code blocks for technical examples
- Blockquotes for important callouts or quotes
- Consistent heading hierarchy (H1 for title, H2 for sections, H3 for subsections)

## 🔄 Pull Request Process

1. **Fork the repository** and create your branch from `main`
2. **Make your changes** following the guidelines above
3. **Update the changelog** if making significant additions
4. **Submit a pull request** with:
   - Clear title describing the change
   - Description of what was added/changed and why
   - Links to any related issues

## 📬 Questions?

- Open an issue for questions about contributing
- Tag issues with `question` for general inquiries
- Tag issues with `enhancement` for new feature/content suggestions

## 🤝 Code of Conduct

- Be respectful and constructive
- Assume good intent
- Focus on the content, not the person
- Credit others' work appropriately
